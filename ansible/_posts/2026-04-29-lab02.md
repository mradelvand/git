## Why Ansible for AWS?

This is a question worth answering before you write a single line of YAML.

**Terraform** is excellent for *declaring* what infrastructure should exist (VPCs, subnets, RDS clusters). **Ansible** is excellent for *configuring* what runs on top of that infrastructure (packages, services, files, users). In the real world, most teams use both — Terraform to provision, Ansible to configure.

This project uses Ansible for both provisioning *and* configuration deliberately, because:

- Learning tools that do both gives you a mental model for where each belongs
- Many smaller teams and startups use only Ansible for everything
- AWS module coverage in Ansible is genuinely comprehensive (hundreds of modules)

**When AWS + Ansible makes sense at work:**
- You already run Ansible for on-premise config and want to extend to AWS
- Your team prefers YAML over HCL
- You need configuration management and cloud provisioning in one pipeline
- You're building internal tooling where simplicity beats sophistication

## Part 1 — AWS IAM: Create a Least-Privilege User

### Why Not Use Root?

The root account has unlimited power over your entire AWS account. If its credentials are compromised, an attacker can run thousands of GPU instances for crypto mining, costing thousands of dollars in minutes. Always use IAM users with scoped permissions.

### Step 1: Create the IAM User in AWS Console

1. Log into AWS Console → search **IAM** → click **Users** → **Add users**
2. Username: `ansible-automation`
3. Click **Next**
4. Select **Attach policies directly**
5. For learning, attach **AmazonEC2FullAccess** (covers everything in this project)

> **In production:** Create a custom policy with only the exact permissions needed. For this project that means: `ec2:*` and `iam:CreateAccessKey`. The principle of least privilege is not optional in professional environments.

6. Click **Next** → **Create user**
7. Click on the user → **Security credentials** tab → **Create access key**
8. Select **Command Line Interface (CLI)** → check the acknowledgment → **Next**
9. Click **Create access key**
10. **Download the CSV file now.** You cannot see the secret key again after this screen.

## Part 2 — Secure Credentials with Ansible Vault

### Why Vault? Why Not Just Export Variables?

The lecture showed using `export AWS_ACCESS_KEY_ID=...` in `.bashrc`. This works but has problems:

- Anyone with access to your machine can read `.bashrc`
- Variables appear in `ps aux` output on some systems
- You might accidentally commit `.bashrc` to a dotfiles repo

**Ansible Vault** encrypts your secrets with AES-256. The encrypted file can safely live in version control — only someone with the vault password can read it.

### Step 1: Create the Vault File

```bash
ansible-vault create vault/aws_credentials.yml
```

Ansible will ask you to set a vault password (choose something strong, you'll need it each time you run playbooks). The editor opens — paste this content, substituting your real values from the CSV:

```yaml
# vault/aws_credentials.yml
# This file is AES-256 encrypted. Never commit the plaintext version.
aws_access_key: "AKIAIOSFODNN7EXAMPLE"
aws_secret_key: "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"
aws_region: "us-west-2"
```

Save and close the editor. View what the encrypted file looks like:

```bash
cat vault/aws_credentials.yml
```

Output (yours will differ — this is just illustrative):
```
$ANSIBLE_VAULT;1.1;AES256
66386439653236336462626566653531623430623636343633636534393335623932393864356633
...
```

### Step 2: Create a Vault Password File (For Local Development Only)

To avoid typing the password on every playbook run during development:

```bash
echo "your_vault_password_here" > ~/.ansible_vault_pass
chmod 600 ~/.ansible_vault_pass
```

Add to `.gitignore`:
```bash
echo "~/.ansible_vault_pass" >> .gitignore
```

> **In a CI/CD pipeline (GitHub Actions, Jenkins):** Store the vault password as an environment secret and pass it with `--vault-password-file` or `ANSIBLE_VAULT_PASSWORD_FILE` environment variable. Never hardcode it.

### ✅ Verification — Vault

```bash
# Decrypt and view (should show your plaintext YAML)
ansible-vault view vault/aws_credentials.yml

# Test that encrypted content can be used in a playbook
ansible localhost -e "@vault/aws_credentials.yml" --ask-vault-pass \
  -m debug -a "var=aws_region"
# Should print: "aws_region": "us-west-2"
```

## Part 5 — Phase 1 Playbook: Create Key Pair & Security Group

### Understanding the Flow

Before we can launch EC2 instances, we need two things:

1. **A Key Pair** — AWS stores the public key on the instance. We keep the private key locally. This is the only way to SSH into the instance without a bastion or SSM.
2. **A Security Group** — AWS's stateful firewall. If you don't define specific rules, the default security group blocks all inbound traffic. Our web server needs port 80 and 22 open.

Create `playbooks/provision.yml`:


```yaml
---
# playbooks/provision.yml
# Phase 1 & 2: Create AWS infrastructure resources then launch EC2 instances.
# Runs on localhost — makes API calls to AWS via boto3.

- name: Provision AWS Infrastructure
  hosts: localhost     #run on YOUR machine (not remote servers)
  connection: local    #don’t use SSH, run locally
  gather_facts: false

  # Load our encrypted credentials and group vars
  vars_files:
    - "../vault/aws_credentials.yml"
    - "../group_vars/all.yml"
    
  vars:
    # Make credential variables available to modules
    aws_common: &aws_common
      aws_access_key: "{{ aws_access_key }}"
      aws_secret_key: "{{ aws_secret_key }}"
      region: "{{ aws_region }}"

  tasks:

    # ─────────────────────────────────────────────────────────────────────────
    # PHASE 1: Key Pair
    # ─────────────────────────────────────────────────────────────────────────

    - name: Create EC2 Key Pair
      amazon.aws.ec2_key:
        <<: *aws_common
        name: "{{ ec2_key_name }}"
        state: present
      register: key_result

    - name: Debug - Show key creation result
      ansible.builtin.debug:
        msg: "Key '{{ ec2_key_name }}' — changed: {{ key_result.changed }}"

    - name: Save private key to disk (only on first creation)
      ansible.builtin.copy:
        content: "{{ key_result.key.private_key }}"
        dest: "../keys/{{ ec2_key_name }}.pem"
        mode: '0600'
      when: key_result.changed
      # Why mode 0600? SSH refuses to use key files readable by other users.
      # chmod 0600 = owner read/write only. This is non-negotiable for SSH.

    - name: Verify key file exists
      ansible.builtin.stat:
        path: "../keys/{{ ec2_key_name }}.pem"
      register: key_file_check

    - name: Assert key file is present
      ansible.builtin.assert:
        that:
          - key_file_check.stat.exists
          - key_file_check.stat.mode == '0600'
        fail_msg: "Key file missing or has wrong permissions!"
        success_msg: "Key file present with correct permissions."

    # ─────────────────────────────────────────────────────────────────────────
    # PHASE 2: Security Groups
    # ─────────────────────────────────────────────────────────────────────────
    # Production note: Use separate security groups per tier (web, app, db).
    # Each group only allows traffic from the group that needs to reach it.
    # Here we create two groups to demonstrate the pattern.

    - name: Create security group for web servers
      amazon.aws.ec2_security_group:
        <<: *aws_common
        name: "ansible-demo-web-sg"
        description: "Web server security group - managed by Ansible"
        tags:
          Name: "ansible-demo-web-sg"
          "{{ ec2_tags | dict2items | map(attribute='key') | first }}": "{{ ec2_tags | dict2items | map(attribute='value') | first }}"
        rules:
          # SSH access — in production restrict this to your office IP or VPN CIDR
          - proto: tcp
            ports:
              - 22
            cidr_ip: 0.0.0.0/0
            rule_desc: "SSH access"
          # HTTP — public web traffic
          - proto: tcp
            ports:
              - 80
            cidr_ip: 0.0.0.0/0
            rule_desc: "HTTP web traffic"
          # HTTPS — public web traffic
          - proto: tcp
            ports:
              - 443
            cidr_ip: 0.0.0.0/0
            rule_desc: "HTTPS web traffic"
        rules_egress:
          # Allow all outbound — standard for internet-facing servers
          - proto: all
            cidr_ip: 0.0.0.0/0
            rule_desc: "All outbound"
        state: present
      register: web_sg_result

    - name: Create security group for app servers
      amazon.aws.ec2_security_group:
        <<: *aws_common
        name: "ansible-demo-app-sg"
        description: "App server security group - managed by Ansible"
        rules:
          # SSH only
          - proto: tcp
            ports:
              - 22
            cidr_ip: 0.0.0.0/0
            rule_desc: "SSH access"
          # Application port — only accessible from within VPC
          - proto: tcp
            ports:
              - 8080
            cidr_ip: 10.0.0.0/8
            rule_desc: "App port - internal VPC only"
        rules_egress:
          - proto: all
            cidr_ip: 0.0.0.0/0
        state: present
      register: app_sg_result

    - name: Print Security Group IDs
      ansible.builtin.debug:
        msg:
          - "Web SG ID : {{ web_sg_result.group_id }}"
          - "App SG ID : {{ app_sg_result.group_id }}"

    # ─────────────────────────────────────────────────────────────────────────
    # PHASE 3: Launch EC2 Instances
    # ─────────────────────────────────────────────────────────────────────────

    - name: Launch Web Server EC2 instance
      amazon.aws.ec2_instance:
        <<: *aws_common
        name: "ansible-demo-webserver"
        key_name: "{{ ec2_key_name }}"
        instance_type: "{{ ec2_instance_type }}"
        image_id: "{{ ec2_ami_id }}"
        security_group: "ansible-demo-web-sg"
        network:
          assign_public_ip: true
        wait: true
        wait_timeout: 300
        exact_count: 1
        state: running
        tags:
          Name: "ansible-demo-webserver"
          Role: "webserver"
          Environment: "dev"
          Project: "ansible-aws-demo"
          ManagedBy: "Ansible"
      register: web_instance_result

    - name: Launch App Server EC2 instance
      amazon.aws.ec2_instance:
        <<: *aws_common
        name: "ansible-demo-appserver"
        key_name: "{{ ec2_key_name }}"
        instance_type: "{{ ec2_instance_type }}"
        image_id: "{{ ec2_ami_id }}"
        security_group: "ansible-demo-app-sg"
        network:
          assign_public_ip: true #gives temporary public IP,Free(while running)
        wait: true
        wait_timeout: 300
        exact_count: 1
        state: running
        tags:
          Name: "ansible-demo-appserver"
          Role: "appserver"
          Environment: "dev"
          Project: "ansible-aws-demo"
          ManagedBy: "Ansible"
      register: app_instance_result

    - name: Print instance public IPs
      ansible.builtin.debug:
        msg:
          - "Web Server IP : {{ web_instance_result.instances[0].public_ip_address }}"
          - "App Server IP : {{ app_instance_result.instances[0].public_ip_address }}"

    - name: Wait for SSH to be available on Web Server
      ansible.builtin.wait_for:
        host: "{{ web_instance_result.instances[0].public_ip_address }}"
        port: 22
        delay: 10
        timeout: 120
        state: started
      delegate_to: localhost

    - name: Wait for SSH to be available on App Server
      ansible.builtin.wait_for:
        host: "{{ app_instance_result.instances[0].public_ip_address }}"
        port: 22
        delay: 10
        timeout: 120
        state: started
      delegate_to: localhost

    - name: Print completion summary
      ansible.builtin.debug:
        msg:
          - "═══════════════════════════════════════════"
          - "  Infrastructure provisioning complete!"
          - "  Web Server: {{ web_instance_result.instances[0].public_ip_address }}"
          - "  App Server: {{ app_instance_result.instances[0].public_ip_address }}"
          - "  Key file  : ./keys/{{ ec2_key_name }}.pem"
          - "═══════════════════════════════════════════"
```

## Part 6 — Phase 2 Playbook: Launch EC2 Instances

> This phase is already included in `provision.yml` above (the "Phase 3: Launch EC2 Instances" section). We covered it together because key pair, security group, and instance creation form one logical unit — you rarely do one without the others.

**Understanding `exact_count: 1` — the most important parameter:**

```yaml
exact_count: 1
```

This tells Ansible "there should be exactly 1 instance matching this name/tags." Without it:

- First run → 1 instance ✅
- Second run → 2 instances 😱
- Third run → 3 instances 😱😱

`exact_count` makes the module **idempotent** — the core Ansible principle. Running the playbook 10 times should produce the same result as running it once.

**Understanding `wait: true`:**

```yaml
wait: true
wait_timeout: 300
```

Without this, Ansible submits the launch request and immediately moves to the next task. The instance isn't ready yet. `wait: true` makes Ansible poll AWS until the instance reaches the `running` state before continuing. The subsequent `wait_for` port 22 check adds a second layer — even when AWS reports `running`, SSH daemon may not be ready yet.

## Part 7 — Dynamic Inventory: Let Ansible Find Your Servers

### The Problem with Static Inventory

If you hard-code IP addresses in an `inventory/hosts.ini` file, you have to update it every time an instance restarts (public IPs change on stop/start), scales up, or is replaced. In any real cloud environment this becomes unmanageable within weeks.

### The Solution: Dynamic Inventory Plugin

The `amazon.aws.aws_ec2` plugin makes live API calls to AWS at playbook run time and builds the inventory dynamically based on running instances and their tags.

Create `inventory/aws_ec2.yml`:


```yaml
# inventory/aws_ec2.yml
# Dynamic inventory configuration for AWS EC2.
# File name MUST end with aws_ec2.yml or aws_ec2.yaml

plugin: amazon.aws.aws_ec2

# AWS credentials (will use environment variables or AWS CLI profile if not specified)
# Better practice: rely on AWS_PROFILE or IAM role rather than hardcoding here
aws_access_key: "{{ lookup('env', 'AWS_ACCESS_KEY_ID') }}"
aws_secret_key: "{{ lookup('env', 'AWS_SECRET_ACCESS_KEY') }}"

regions:
  - us-west-2

# Only show running instances that belong to our project
filters:
  instance-state-name: running
  "tag:Project": "ansible-aws-demo"

# Create Ansible host groups from EC2 tags
# This is the key feature: tags become group names!
keyed_groups:
  # Creates groups like: tag_Role_webserver, tag_Role_appserver
  - key: tags.Role
    prefix: tag_Role
  # Creates groups like: tag_Environment_dev, tag_Environment_prod
  - key: tags.Environment
    prefix: tag_Environment

# How to connect to each instance
compose:
  # Use public IP for SSH (private IP for VPN/bastion setups)
  ansible_host: public_ip_address
  # Amazon Linux uses ec2-user, Ubuntu uses ubuntu, CentOS uses centos
  ansible_user: "'ec2-user'"
  ansible_ssh_private_key_file: "'./keys/ansible-key.pem'"

# Strict mode: fail if any host variable lookup fails
strict: false
```

## Part 9 — Teardown: Destroy Everything Cleanly

**Always run this when done.** Leaving EC2 instances running costs money.

Create `playbooks/teardown.yml`:


```yaml
---
# playbooks/teardown.yml
# Destroy ALL resources created by this project.
# Prompts for confirmation before doing anything destructive.

- name: Teardown AWS Infrastructure
  hosts: localhost
  connection: local
  gather_facts: false

  vars_files:
    - "../vault/aws_credentials.yml"

  vars:
    aws_common: &aws_common
      aws_access_key: "{{ aws_access_key }}"
      aws_secret_key: "{{ aws_secret_key }}"
      region: "{{ aws_region }}"

  tasks:

    - name: Safety prompt — confirm teardown
      ansible.builtin.pause:
        prompt: |
          ⚠️  WARNING: This will PERMANENTLY DELETE:
            - EC2 instances (ansible-demo-webserver, ansible-demo-appserver)
            - Security Groups (ansible-demo-web-sg, ansible-demo-app-sg)
            - Key Pair (ansible-key)
            - Local key file (./keys/ansible-key.pem)

          Type YES to continue, anything else to abort
      register: confirm

    - name: Abort if not confirmed
      ansible.builtin.fail:
        msg: "Teardown aborted by user."
      when: confirm.user_input != "YES"

    - name: Terminate Web Server EC2 instance
      amazon.aws.ec2_instance:
        <<: *aws_common
        filters:
          "tag:Name": "ansible-demo-webserver"
          "tag:Project": "ansible-aws-demo"
        state: absent
        wait: true

    - name: Terminate App Server EC2 instance
      amazon.aws.ec2_instance:
        <<: *aws_common
        filters:
          "tag:Name": "ansible-demo-appserver"
          "tag:Project": "ansible-aws-demo"
        state: absent
        wait: true

    - name: Wait for instances to be fully terminated
      ansible.builtin.pause:
        seconds: 15
        prompt: "Waiting for instances to terminate before deleting security groups..."

    - name: Delete Web Security Group
      amazon.aws.ec2_security_group:
        <<: *aws_common
        name: "ansible-demo-web-sg"
        state: absent

    - name: Delete App Security Group
      amazon.aws.ec2_security_group:
        <<: *aws_common
        name: "ansible-demo-app-sg"
        state: absent

    - name: Delete EC2 Key Pair from AWS
      amazon.aws.ec2_key:
        <<: *aws_common
        name: "{{ ec2_key_name }}"
        state: absent

    - name: Remove local key file
      ansible.builtin.file:
        path: "./keys/{{ ec2_key_name }}.pem"
        state: absent

    - name: Teardown complete
      ansible.builtin.debug:
        msg:
          - "All resources deleted."
          - "Verify in AWS Console that no instances remain."
```


```bash
ansible-playbook playbooks/teardown.yml
```

**Talk about these specifically:**

## Troubleshooting Reference

### Error: `No module named boto3`

```bash
pip install boto3 botocore
# Or for Python3 specifically:
pip3 install boto3 botocore
```

### Error: `You must specify a region`

```bash
# Either add to the playbook task:
region: us-west-2

# Or export:
export AWS_DEFAULT_REGION=us-west-2

# Or add to ~/.bashrc for persistence
```

### Error: `Unable to locate credentials`

```bash
# Check environment variables are set
echo $AWS_ACCESS_KEY_ID  # Should print your key ID (not blank)

# Or use AWS CLI profile
aws configure --profile ansible
# Then in ansible.cfg or inventory, set: aws_profile: ansible
```

### Error: `InvalidAMIID.NotFound`

AMI IDs are **region-specific**. The AMI `ami-02d8bad0a1da4b6fd` only works in `us-west-2`. Find the correct AMI for your region:

```bash
aws ec2 describe-images \
  --owners amazon \
  --filters "Name=name,Values=al2023-ami-*" \
            "Name=architecture,Values=x86_64" \
  --query 'Images | sort_by(@, &CreationDate) | [-1].ImageId' \
  --output text \
  --region us-east-1
```

### Error: `An error occurred (DependencyViolation) when deleting security group`

You must terminate instances before deleting their security group. The teardown playbook handles this with `wait: true` on instance termination. If running manually, wait ~60 seconds between the two operations.

### Dynamic inventory returns no hosts

```bash
# Debug the inventory with verbose output
ansible-inventory -i inventory/aws_ec2.yml --list -vvv

# Check your filter tags match what's on the instances
aws ec2 describe-instances \
  --filters "Name=instance-state-name,Values=running" \
  --query 'Reservations[*].Instances[*].Tags' \
  --output json
```

### SSH: Permission denied (publickey)

```bash
# Check key file permissions
ls -la keys/ansible-key.pem  # Must be 0600
chmod 600 keys/ansible-key.pem

# Check the correct username for your AMI
# Amazon Linux: ec2-user
# Ubuntu: ubuntu
# CentOS: centos or ec2-user
# Debian: admin

# Test SSH manually
ssh -i keys/ansible-key.pem -v ec2-user@<instance-ip>
```
